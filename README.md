Разработка концептуально-логических моделей базы данных бизнес-процесса
Описание проекта
Данный проект представляет собой курсовую работу по дисциплине «Базы данных», выполненную студентом 2-го курса института информатики, математики и робототехники Уфимского университета науки и технологий (УУНИТ) по направлению подготовки «Программная инженерия». Автором работы является Колобников Богдан. Темой работы является разработка концептуально-логических моделей базы данных для автоматизации бизнес-процесса приемной комиссии вуза. В рамках проекта автоматизируются три ключевые функции: прием заявлений и документов от абитуриентов, зачисление абитуриентов по общему конкурсу, а также рассмотрение апелляций по приему документов и результатам зачисления.

Структура и содержание работы
Работа включает в себя пять основных этапов проектирования, каждый из которых последовательно развивает модели данных от внешнего представления до внутренней реляционной структуры. На первом этапе разработаны внешние иерархические модели для каждой автоматизируемой функции. Для функции приема заявлений создана модель с сущностью «Направление», содержащей вложенные агрегаты выбранных абитуриентов и их документов. Для функции зачисления разработана модель с сущностью «Категория абитуриентов» и вложенным агрегатом зачисленных абитуриентов с их льготами. Для функции апелляций построена модель с сущностью «Апеллянт», включающей агрегаты апелляций и присутствия на заседаниях. Для каждой модели определены ограничения атрибутов, кортежей, уникальности, а также операционные правила поддержания целостности данных.

На втором этапе выполнена нормализация локальных ER-моделей. Исходные иерархические модели преобразованы в третью нормальную форму путем вычленения многозначных и однозначных скрытых сущностей. В результате для функции приема заявлений получена модель из семи сущностей, включающая «Направление», «Абитуриент», «Документ абитуриента», «Направление абитуриента» и другие. Для функции зачисления разработана модель из девяти сущностей с выделением справочников льгот и категорий. Для функции апелляций создана модель из восьми сущностей, включающая сущности «Апелляция», «Заседание», «Участие в заседании» и справочные сущности. На этом этапе также сформулированы правила ссылочной целостности, определяющие поведение системы при удалении экземпляров родительских сущностей.

Третий этап посвящен синтезу глобальной ER-модели, объединяющей все три локальные модели в единое целое. В ходе анализа дублирования выявлены эквивалентные сущности: «Абитуриент» из функций приема и зачисления, «Апеллянт» из функции апелляций, а также «Секретарь», «Руководитель» и «Член комиссии», объединенные в суперсущность «Сотрудник». Для унификации справочных данных создана суперсущность «Справочник» с категориями «Вид документа», «Категория абитуриента», «Льгота» и «Вид претензии». Устранены дублирующиеся атрибуты и связи, а атрибуты-ссылки заменены явными связями между сущностями. Итоговая глобальная модель включает 15 сущностей в третьей нормальной форме.

На четвертом этапе выполнено проектирование внутренней реляционной модели в двух вариантах: на основе натуральных ключей и на основе суррогатных ключей. Модель на натуральных ключах использует составные ключи из предметной области, такие как сочетание кода направления и факультета, что обеспечивает естественную идентификацию записей. Модель на суррогатных ключах вводит автоинкрементные идентификаторы для каждой таблицы, что упрощает ссылочную целостность и повышает производительность запросов. Для обеих моделей разработаны спецификации ограничений атрибутов, кортежей, уникальности, а также операционные правила и стандартные ссылочные правила, определяющие поведение при каскадном или строгом удалении записей.

Заключительный пятый этап посвящен разработке моделей отображения, связывающих внешние иерархические модели с внутренней реляционной моделью. Созданы представления и хранимые процедуры, реализующие иерархические структуры данных, требуемые для каждой автоматизируемой функции. Для функции приема заявлений разработано представление «вирт_Направление_предварительно», отображающее направления подготовки с их факультетами, хранимая процедура «проц_Выбрано_абитуриентом_предварительно» для получения списка абитуриентов по конкретному направлению, а также процедура «проц_Документ_абитуриента» для получения документов каждого абитуриента. Дополнительно создано представление «вирт_Направление_предварительно_Кол-во» для подсчета количества направлений по факультетам. Все представления и процедуры построены на основе эквисоединений сущностей внутренней реляционной модели с использованием фильтрации и агрегатных функций.

Техническая реализация
Проект реализован с использованием СУБД MySQL и MariaDB. Для графического моделирования применялся Microsoft Office Visio. В составе проекта представлены четыре SQL-скрипта: скрипт создания реляционной модели на натуральных ключах, скрипт создания реляционной модели на суррогатных ключах, шаблон моделей отображения и финальный скрипт с реализованными представлениями и хранимыми процедурами. Разработанная база данных обеспечивает полное покрытие бизнес-процесса приемной комиссии, включая регистрацию абитуриентов и их документов, учет выбора направлений подготовки, формирование приказов о зачислении, учет льгот, а также регистрацию и рассмотрение апелляций.

Результаты работы
В результате выполнения курсовой работы спроектирована полноценная база данных для автоматизации работы приемной комиссии вуза. Разработанные модели охватывают все этапы взаимодействия абитуриентов с приемной комиссией: от подачи документов до зачисления и возможного апелляционного обжалования. Итоговая реляционная модель включает 15 таблиц, объединенных системой внешних ключей и ограничений целостности. Модели отображения обеспечивают удобный доступ к данным в формате, соответствующем иерархическим представлениям, что упрощает разработку прикладного программного обеспечения для автоматизации работы приемной комиссии.
